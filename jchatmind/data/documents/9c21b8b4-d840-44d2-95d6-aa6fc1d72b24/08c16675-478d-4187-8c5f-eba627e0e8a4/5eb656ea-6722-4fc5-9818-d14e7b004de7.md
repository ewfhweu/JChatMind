# 理论
## 模型与能力
大模型，Agent,Agentic（多Agent协作）,工具调用，MCP	,向量模型，RAG（记忆+知识库）

| Agent | Agentic |
| --- | --- |
| 单个智能体 | 多个智能体 |
| 一个角色 | 多角色协作 |
| 简单任务 | 复杂系统级任务 |


工具调用，让模型能够真正的动手，因为LLM不能访问数据库，不呢个发HTTP，不能查询redis，不能写问文件

MCP：工具调用的统一协议,给大模型的USB接口标准

RAG: 先查资料，再让模型基于资料说话  

用户问题->向量检索（查资料）->把资料塞进 Prompt->LLM 回答

1. 多模态模型：可以处理的不仅仅是文本，还可以处理图片，音频，视频，本质上是模型能够处理的输入和输出形式变多了，
2. 向量模型：不负责生成内容，向量模型输出是一段文本在语义空间的位置，两段内容在语义层面有相似之处，向量模型不处理答案，只负责找到该看的内容
3. 大语言模型：负责理解自然语言

```plain
用户请求
  ↓
Agent（目标驱动）
  ↓
Think（需要知识？）
  ↓
RAG（向量检索）
  ↓
Think（需要行动？）
  ↓
Tool / MCP（真实调用系统）
  ↓
反馈
  ↓
下一轮 Think

```

## Agent-一次性和多步骤任务
模型代表的只是能力，而不是做事的主体，大部分AI并不是一问一答的处理方式，需要多次execute才能处理成功这个任务

+ 一次性任务不依赖中间状态和多步推进，输入和输出是强因果关系
+ 多步骤任务有明确的最终目标，需要分步骤完成，每一步都依赖前一步的结果，中间过程必须和真实系统交互

Agent是一个带着目标，会分步骤推进任务的AI行为单元

Agent = LLM

      + Goal（目标）

      + Memory（上下文/状态）

      + Tools（工具）

### Agent和Workflow的本质区别
Agent:智能体，模型对过程拥有决策权，根据当前状态，决定下一步行动，整个任务推进过程都是动态的

Agent：目标，状态，行动

Workflow:工作流,整个执行过程都是提前设计好的，每一步做什么，都是固定的，模型只是节点被调用

## Agent如何真正做事
模型负责判断和决策，系统负责真实执行

当AI按照约定，返回工具名称和参数结构后，系统可以解析AI的返回和调用真实的函数和接口，获取执行结果，再讲结果原样交还给AI

从项目内部的函数变成可被AI统一使用的外部能力

MCP关注能力如何被调用和管理,关注当前的调用工具

### 模型调度的核心-chatclientregistry
当系统存在多个chatclient后，我们需要设置模型调度，根据Agent的配置，选择正确的模型

<!-- 这是一张图片，ocr 内容为： -->
![](https://cdn.nlark.com/yuque/0/2025/png/40454691/1766932118383-702ad6b4-0269-49ac-803b-6108c716f238.png)

```plain
@Component
public class ChatClientRegistry {

    private final Map<String, ChatClient> chatClients;

    public ChatClientRegistry(Map<String, ChatClient> chatClients) {
        this.chatClients = chatClients;
    }

    public ChatClient get(String key) {
        return chatClients.get(key);
    }
}
```

这部分代码解决了新增模型，不需要更改任何调度代码，并且模型注册是声明式的，不是控制流式的，模型选择完全由数据驱动

用注册表模式做多模型支持
